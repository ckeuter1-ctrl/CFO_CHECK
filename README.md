# Лид-магнит «10 признаков теневой Excel-системы»

Production-ready MVP на Next.js App Router для сбора лида, выдачи интерактивного чек-листа и передачи событий в существующий лид Битрикс24.

## Что реализовано

- лёгкая форма: обязательный рабочий email, необязательные имя, компания и роль;
- чтение `utm_source`, `utm_medium`, `utm_campaign`, `utm_content` из URL с fallback `direct`;
- server-side создание лида через `crm.lead.add.json` и `FormData`;
- сохранение `leadId` и email в `localStorage`;
- восстановление thank-you экрана после обновления страницы без создания нового лида;
- интерактивный чек-лист, динамический скоринг и сохранение отмеченных пунктов;
- debounce score update на 1000 мс;
- обновление того же лида для открытия чек-листа, выбора фокуса и запроса диагностики;
- защита кнопки диагностики и события открытия от повторной отправки;
- debug-блок только при наличии `?debug=1`;
- печать / сохранение чек-листа в PDF средствами браузера.

## Локальный запуск

Требования: Node.js 20.9+ и npm.

```bash
npm install
cp .env.example .env.local
npm run dev
```

Откройте [http://localhost:3000](http://localhost:3000).

Для production-проверки:

```bash
npm run lint
npm run typecheck
npm run build
npm start
```

## ENV

Единственная обязательная серверная переменная:

```env
BITRIX_WEBHOOK_BASE_URL=https://your-domain.bitrix24.ru/rest/USER_ID/WEBHOOK_CODE/
```

URL должен быть **базовым**, с завершающим `/` и без метода `crm.*`. Приложение само добавляет методы:

- `crm.lead.add.json`;
- `crm.lead.get.json`;
- `crm.lead.update.json`;
- `crm.timeline.comment.add.json`.

> Webhook нельзя хранить во frontend-коде, добавлять в переменные с префиксом `NEXT_PUBLIC_`, показывать в UI или коммитить в Git. `.env.local` исключён через `.gitignore`.

Перед production-развертыванием обязательно создайте **новый webhook** Битрикс24 с минимально необходимыми правами CRM. Не используйте webhook, который когда-либо публиковался в переписке, документации, логах или истории Git.

## Настройка Vercel

1. Импортируйте GitHub-репозиторий в Vercel.
2. Откройте **Project Settings → Environment Variables**.
3. Добавьте `BITRIX_WEBHOOK_BASE_URL`.
4. Выберите окружения `Production`, `Preview` и, если нужно, `Development`.
5. Сохраните переменную и сделайте новый deploy.
6. Не создавайте `NEXT_PUBLIC_BITRIX_WEBHOOK_BASE_URL`: любой `NEXT_PUBLIC_*` секрет попадёт в браузерный bundle.

## Настройка и проверка Битрикс24

Webhook должен иметь доступ к CRM-лидам и таймлайну. После настройки ENV:

1. Откройте страницу с тестовой UTM-ссылкой.
2. Отправьте форму один раз.
3. В Битрикс24 найдите лид с заголовком `Лид-магнит: 10 признаков теневой Excel-системы`.
4. Проверьте email, имя, компанию, должность, стандартные UTM-поля и `COMMENTS`.
5. Откройте чек-лист, выберите фокус, измените score и запросите диагностику.
6. Убедитесь, что события появились в таймлайне того же лида, а после диагностики заголовок изменился на `MQL: запрос диагностики — 10 признаков теневой Excel-системы`.

Для просмотра клиентского состояния можно временно добавить `?debug=1`. Без этого параметра debug-блок не рендерится. Webhook в debug не выводится.

## UTM-ссылки

Подставьте нужный origin перед query string.

- VC: `?utm_source=vc&utm_medium=article&utm_campaign=excel_shadow&utm_content=checklist_cta`
- Telegram: `?utm_source=telegram&utm_medium=social&utm_campaign=excel_shadow&utm_content=teaser`
- LinkedIn: `?utm_source=linkedin&utm_medium=social&utm_campaign=excel_shadow&utm_content=teaser`

Если параметр отсутствует или пуст, его значение отправляется как `direct`.

## Acceptance criteria

### 1. Создание одного лида

Открыть:

```text
/?utm_source=vc&utm_medium=article&utm_campaign=excel_shadow&utm_content=checklist_cta
```

Заполнить:

- email: `test@example.com`;
- имя: `Александр CFO`;
- компания: `Капитал Ритейл Групп`;
- роль: `Руководитель департамента финансового анализа`.

Ожидание: в Битрикс24 создан ровно один лид с контактными данными и UTM.

### 2. Открытие чек-листа

Нажать «Открыть / скачать чек-лист» три раза.

Ожидание: новые лиды не создаются; событие относится к исходному лиду и дедуплицируется в браузере.

### 3. Выбор фокуса

Выбрать «Управленческий отчет».

Ожидание: в исходном лиде появляется timeline comment, а фокус добавляется в `COMMENTS`; новый лид не создаётся.

### 4. Обновление score

Быстро отметить три чекбокса и подождать более одной секунды.

Ожидание: отправляется одно debounced-событие со score `3` в исходный лид; новый лид не создаётся.

### 5. Запрос диагностики

Нажать «Запросить диагностику».

Ожидание: исходный лид получает комментарий и новый title, кнопка блокируется и показывает «Заявка принята. Свяжемся с вами.»; повторный клик ничего не отправляет.

### 6. Проверка после обновления

Обновить страницу и повторить действия.

Ожидание: thank-you экран и чек-лист восстанавливаются по `excel_shadow_lead_id`; новая форма не отправляется и новый лид не создаётся. Состояние диагностики, выбранного фокуса и score сохраняется для этого `leadId`.

## Ограничения MVP

Дедупликация реализована в рамках конкретного браузера через `localStorage`, как предусмотрено требованиями MVP. Очистка storage, другой браузер или другое устройство считаются новой клиентской сессией. Для глобальной дедупликации по email на следующем этапе потребуется серверное хранилище или согласованный поиск существующего лида в CRM до создания.
